# Step 4: Static export + project structure

Lock in two foundational decisions before we start writing the data layer:

1. **Build target** ‚Äî switch Next.js into static-export mode so the final artifact is a folder of HTML/CSS/JS we can drop on Cloudflare Pages, GitHub Pages, or any CDN.
2. **Folder layout** ‚Äî agree on where conferences data, build-time scripts, and shared types live, so later steps don't sprawl.

Doing this *before* coding features means we won't accidentally reach for `fetch` in a Server Component or a Route Handler ‚Äî both of which static export silently disables.

---

## 1. Enable static export

### 1.1 Edit `next.config.ts`

```ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  output: "export",
  images: {
    unoptimized: true,
  },
  trailingSlash: true,
};

export default nextConfig;
```

| Option | Why |
|---|---|
| `output: "export"` | Tells Next.js to emit a static site to `out/` on `pnpm build`. No Node server, no SSR at runtime ‚Äî everything is pre-rendered at build time. |
| `images.unoptimized: true` | The built-in `next/image` optimizer needs a Node runtime. With export it must be off or replaced. We don't need it for this site (we'll use SVG conference logos / Lucide icons). |
| `trailingSlash: true` | Generates `out/about/index.html` instead of `out/about.html`. Plays nice with most static hosts (GH Pages, Cloudflare Pages) ‚Äî no host-specific rewrite rules needed. |

### 1.2 What you can no longer use

Static export = build-time only. The following are now **compile errors or silent no-ops**:

| Feature | Status under `output: "export"` |
|---|---|
| Route Handlers (`app/api/.../route.ts`) | ‚ùå build error |
| Server Actions (`"use server"` functions) | ‚ùå build error |
| `revalidate` / `revalidatePath` | ‚ùå ignored |
| Middleware (`middleware.ts`) | ‚ùå ignored |
| Dynamic routes without `generateStaticParams` | ‚ùå build error |
| `cookies()`, `headers()`, `draftMode()` | ‚ùå runtime undefined |
| `next/image` with default loader | ‚öÝÔ∏è needs `unoptimized` or custom loader |

What still works (and is all we need):

- React Server Components rendered **at build time**
- `generateStaticParams` for conference detail pages (Step 8)
- Client Components (`"use client"`) for interactivity
- File-system imports of JSON / YAML at build time
- `next/font`, `next/script`, Tailwind, shadcn ‚Äî all unaffected

> Mental model: think Astro / Hugo with React syntax. The site that ships is a folder of files.

## 2. Define the folder layout

We're adding three new top-level folders. Create them and seed each with a `.gitkeep` so git tracks the empty dirs:

```bash
mkdir -p data scripts types
touch data/.gitkeep scripts/.gitkeep types/.gitkeep
```

Final layout ‚Äî each folder has one job, no overlap:

### Boundary rules (worth memorizing)

| If the code‚Ä¶ | Lives in |
|---|---|
| has JSX | `app/` (route) or `components/` (reusable) |
| is a pure function with no React import | `lib/` |
| is a Zod schema or shared type | `types/` |
| runs only at build time (Node APIs OK) | `scripts/` |
| is hand-curated raw data | `data/*.yml` |
| is generated data | `data/*.json` (and in `.gitignore` if regenerated on every build) |

> We **commit** `data/conferences.json` for now (so the repo builds without running the sync first). Step 13's GitHub Action will keep it fresh.

## 3. Verify the build

Run a full production build to confirm export mode works:

```bash
pnpm build
```

You should see, near the end:

```
‚úì Generating static pages (5/5)
‚úì Exporting (3/3)

Route (app)                Size  First Load JS
‚îå ‚óã /                      ...   ...
‚îî ‚óã /_not-found            ...   ...
‚óã  (Static)  prerendered as static content
```

Check the output:

```bash
ls out/
```

Expected:

```
404.html  _next/  favicon.ico  index.html  ...
```

Smoke-test the static bundle locally ‚Äî just serve `out/` with any static server. Easiest one-liner:

```bash
pnpm dlx serve out
```

Open the printed URL (usually <http://localhost:3000>). You should see the same smoke-test page from Step 3 ‚Äî three clocks ticking, theme toggle working.

> Why `serve` and not `pnpm start`? `next start` requires the build to **not** be `export` mode (it expects a Node server). For exported sites, any static server works (`serve`, `http-server`, `python -m http.server`, nginx, ...).

Stop with `Ctrl+C`.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `Error: Image Optimization using the default loader is not compatible with \`output: 'export'\`` | Forgot `images.unoptimized: true` | Add it to `next.config.ts` |
| `Page "/foo/[slug]" is missing "generateStaticParams()"` | Dynamic route with no params provider | Either add `generateStaticParams` (Step 8) or delete the route |
| `pnpm dev` works but `pnpm build` complains about Route Handlers / Server Actions | You added an API route after enabling export | Move the logic to `scripts/` (build-time) or a Client Component (runtime) |
| `out/` HTML loads but assets 404 | Hosted under a subpath (e.g. GH Pages project site) | Set `basePath: "/arc-conf"` in `next.config.ts` for that deploy target (Step 14) |
